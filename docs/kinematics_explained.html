🤖 Kinematics Explained: How Your Robot Moves

A simple explanation of what happens between "port detected" and "robot moves."

🎯 The Problem

You have a camera. It sees a port at pixel position (cx, cy). The port should be at the center of the image. How does the robot move to center it?

📷 Camera View

Image: 640 x 480
+
Center (320, 240)
●
Port detected (340, 260)
↙ 20px error

📊 Calculation

Port position: (340, 260)

Image center: (320, 240)

error_x = 340 - 320 = +20 pixels

error_y = 260 - 240 = +20 pixels

Port is:

  • 20 pixels RIGHT of center
  • 20 pixels BELOW center

Robot must move:

  • LEFT to center horizontally
  • FORWARD to center vertically

Hand Position

Joint Angles

Forward Kinematics

Question: "Given joint angles, where is the robot's hand?"

INPUT:  Joint angles: q₁=0.5, q₂=-1.2, q₃=0.8, q₄=-0.3, q₅=1.5, q₆=0

OUTPUT: Hand position: x=0.35m, y=-0.1m, z=0.5m (pointing down)

Inverse Kinematics

Question: "Given desired hand position, what joint angles do I need?"

INPUT:  I want hand at: x=0.3m, y=0.0m, z=0.4m

OUTPUT: Joint angles: q₁=0.2, q₂=-1.5, q₃=1.0, q₄=..., q₅=..., q₆=...
Inverse Kinematics is harder!
There can be multiple solutions (elbow up vs elbow down), or no solution if unreachable.

📐 The Jacobian (Simplified)

The Jacobian is a matrix that tells you: "If I rotate this joint, how fast does the hand move?"

Hand Velocity = Jacobian × Joint Velocities

v = J × q̇

Think of it This Way

  • Each column of the Jacobian = "how one joint affects the hand"
  • If you know joint velocities → multiply by Jacobian → get hand velocity
  • If you know hand velocity → multiply by inverse Jacobian → get joint velocities
Key insight: The Jacobian changes as the robot moves. It depends on current joint angles.

✅ What Your Code Does

You use: Cartesian Velocity Control
The controller handles Jacobian and inverse kinematics FOR YOU.

Step 1: Get Pixel Error

Camera detects port at (cx, cy) = (340, 260)
Image center is at         (w/2, h/2) = (320, 240)

error_x = 340 - 320 = +20 pixels  (port is RIGHT of center)
error_y = 260 - 240 = +20 pixels  (port is BELOW center)

Step 2: Convert to Desired Velocity

gain = 0.0003  (converts pixels to meters/second)

vx = gain × error_y = 0.0003 × 20 = 0.006 m/s  (move forward)
vy = gain × error_x = 0.0003 × 20 = 0.006 m/s  (move left)

Why?
  - Port BELOW center → robot moves FORWARD
  - Port RIGHT of center → robot moves LEFT

Step 3: Send to Controller

msg = MotionUpdate()
msg.velocity.linear.x = 0.006   # Forward at 0.006 m/s
msg.velocity.linear.y = 0.006   # Left at 0.006 m/s
msg.velocity.linear.z = 0.0     # Don't move up/down
msg.velocity.angular = (0, 0, 0)  # Don't rotate

publish(msg)

Step 4: Controller Does the Math (Hidden)

You don't see this. The controller does it automatically.

Controller reads: current joint positions q
Controller computes: Jacobian J(q)
Controller computes: pseudo-inverse J⁺
Controller solves: q̇ = J⁺ × [vx, vy, vz, 0, 0, 0]ᵀ
Controller sends: joint velocities to robot
What you send: "Move at 0.006 m/s in X, 0.006 m/s in Y"
What controller does: Computes Jacobian, solves inverse kinematics, sends joint commands

❌ The Alternative: Compute Jacobian Yourself

You could compute the Jacobian yourself and send joint velocities directly. This is more complex.

What That Would Look Like

# Step 1-2: Same as before (pixel error → velocity)

# Step 3: Get current joint positions
q = [q₁, q₂, q₃, q₄, q₅, q₆] = read_joint_states()

# Step 4: Compute Jacobian (YOUR CODE)
J = compute_jacobian(q)  # 6×6 matrix using DH parameters

# Step 5: Compute pseudo-inverse (YOUR CODE)
J_pinv = pseudo_inverse(J)

# Step 6: Solve for joint velocities (YOUR CODE)
v = [vx, vy, vz, 0, 0, 0]
q_dot = J_pinv @ v  # Matrix multiplication

# Step 7: Send joint velocities
msg = JointMotionUpdate()
msg.velocities = q_dot
publish(msg)

🤔 Is This Standard?

For visual servoing: YES.

There are two types of inverse kinematics:

Type Use Case Method
Position IK "I want to reach point X" Analytical or numerical solver
Velocity IK "I want to move at velocity v" Jacobian pseudo-inverse ← YOU ARE HERE

Why Velocity IK is Right for You

  • Visual servoing is continuous correction, not point-to-point
  • You respond to pixel error in real-time
  • Velocity control naturally handles continuous motion
  • The controller already has a well-tuned implementation
Summary: Your approach sends "move this fast in this direction" and the controller figures out the joints. This is standard for visual servoing.
