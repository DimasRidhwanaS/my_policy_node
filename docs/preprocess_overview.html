🅰️ a_image_processing/

Scripts related to camera and perception.

_1_camera_stream.py IN USE

Purpose

Live camera feed with YOLO detection overlay. Publishes detection results to a ROS topic.

What it does

  1. Subscribes to camera topics: /center_camera/image, /left_camera/image, /right_camera/image
  2. Runs YOLO detection on each frame using model best.pt
  3. Draws bounding boxes, center crosshair, and direction arrows
  4. Publishes detection JSON to /port_detection topic
  5. Shows OpenCV window with keyboard controls

Published Topics

/port_detection

JSON payload:

{
  "port_type": "sfp_port",      // or "sc_port", null if not detected
  "cx": 320.0,                   // center X in pixels
  "cy": 240.0,                   // center Y in pixels
  "confidence": 0.87,
  "img_width": 640,
  "img_height": 480,
  "detected": true
}

Run Command

cd /home/ridhwana/ws_aic/src/aic && pixi run python3 my_policy_node/my_policy_node/preprocess/a_image_processing/_1_camera_stream.py --port sc

Keyboard Controls

1Switch to center camera
2Switch to left camera
3Switch to right camera

Args

FlagValuesDefaultDescription
--portsfp, scNoneFilter detection to specific port type. If omitted, shows all ports.

🅱️ b_move_robot/

Scripts related to robot movement and control.

_0_move_to_init.py IN USE

Purpose

Move robot to home/initial position using joint-space control.

What it does

  1. Reads current joint states from /joint_states
  2. Defines home position (all joints = 0.0)
  3. Sends JointMotionUpdate with target positions to /aic_controller/joint_commands
  4. Waits for robot to reach position

Control Mode

JOINT mode

Uses JointMotionUpdate with target joint positions, stiffness, and damping.

Run Command

cd /home/ridhwana/ws_aic/src/aic && pixi run python3 my_policy_node/my_policy_node/preprocess/b_move_robot/_0_move_to_init.py
Note: This script does NOT call ChangeTargetMode service. Ensure controller is already in JOINT mode, or add the service call.
_1_circle_move.py IN USE

Purpose

Move robot in a circular pattern in the XY plane. Useful for testing Cartesian control.

What it does

  1. Sets controller to CARTESIAN mode via ChangeTargetMode service
  2. Defines circle center, radius, height
  3. Steps through angles (default 30° increments)
  4. Publishes MotionUpdate pose commands to /aic_controller/pose_commands
  5. End effector stays perpendicular (roll=180°, pitch=0, yaw=0)

Parameters (hardcoded)

ParamValueDescription
center_x-0.2Circle center X (meters)
center_y0.0Circle center Y (meters)
radius0.1Circle radius (meters)
height0.3Z height (meters)
angle_step_deg30.0Step size (degrees)

Run Command

cd /home/ridhwana/ws_aic/src/aic && pixi run python3 my_policy_node/my_policy_node/preprocess/b_move_robot/_1_circle_move.py

Control Flow

1. Call ChangeTargetMode → MODE_CARTESIAN
2. For each angle θ in [0°, 30°, 60°, ..., 360°]:
   a. x = center_x + radius * cos(θ)
   b. y = center_y + radius * sin(θ)
   c. z = height
   d. Publish MotionUpdate with pose
   e. Wait 1.5 seconds
_2_move_to_port.py IN USE

Purpose

Continuously center a detected port in the camera frame using Jacobian-based velocity control.

What it does

  1. Subscribes to /port_detection (from _1_camera_stream.py)
  2. Subscribes to /joint_states for current configuration
  3. Subscribes to /aic_controller/controller_state for TCP pose
  4. Computes 6×6 Jacobian at current configuration
  5. Maps desired XY velocity to joint velocities via pseudo-inverse
  6. Maintains Z at target height (0.3m)
  7. Runs control loop at 30Hz
Requires: _1_camera_stream.py must be running! This script does NOT run YOLO itself — it only consumes /port_detection topic.

Subscribed Topics

/port_detection
/joint_states
/aic_controller/controller_state

Published Topics

/aic_controller/joint_commands

Services Called

/aic_controller/change_target_mode

Control Mode

JOINT mode (velocity control)

Run Command

# Terminal 1 - Camera stream (must run first!) cd /home/ridhwana/ws_aic/src/aic && pixi run python3 my_policy_node/my_policy_node/preprocess/a_image_processing/_1_camera_stream.py --port sc # Terminal 2 - Movement cd /home/ridhwana/ws_aic/src/aic && pixi run python3 my_policy_node/my_policy_node/preprocess/b_move_robot/_2_move_to_port.py --port sc

Args

FlagValuesDefaultDescription
--portsfp, scsfpPort type to track. Filters detections to match class (e.g., "sc_port").

Control Algorithm

Loop at 30Hz:
  1. Get detection (cx, cy, img_w, img_h) from /port_detection
  2. Calculate pixel error:
     error_x = cx - img_w/2     # positive = port right of center
     error_y = cy - img_h/2     # positive = port below center

  3. Convert to desired Cartesian velocity:
     vx = sign_y_to_vx * error_y * 0.0003  # vertical error → X motion
     vy = sign_x_to_vy * error_x * 0.0003  # horizontal error → Y motion
     vz = Z correction to maintain height

  4. Compute Jacobian J at current joint config
  5. Solve: q_dot = J_pinv @ [vx, vy, vz, 0, 0, 0]^T
  6. Clamp velocities
  7. Publish JointMotionUpdate with q_dot
Why Jacobian? This is more precise than Cartesian velocity commands because it respects robot kinematics and keeps end effector orientation locked perpendicular.

Step-by-step

  1. Start Gazebo + Robot (eval sim or your launch file)
  2. Start camera stream: _1_camera_stream.py --port sc
  3. Start movement: _2_move_to_port.py --port sc
  4. The robot will continuously track and center the detected port

🔧 Integration into Attempt1.py

The standalone scripts use ROS pub/sub directly. To integrate into Attempt1.py (which uses move_robot() and get_observation() callbacks), you need to:

From _2_move_to_port.py → PortSearch class

Standalone version:                    Policy version:
────────────────────                    ────────────────
self.create_subscription(               → get_observation()
    String, "/port_detection",            (returns Observation with center_image)
    callback
)

self.joint_pub.publish(                 → move_robot(
    JointMotionUpdate(...)                 JointMotionUpdate(...)
)                                        )

self.change_mode_client.call_async(     → Controller mode is set by engine
    ChangeTargetMode.Request()           (or manually via lifecycle)
)

Key Pattern

The logic stays the same — only the I/O changes. Standalone scripts are your test harness. Once verified, copy the core logic into the policy class and swap ROS primitives for callbacks.

📝 Notes

Model Path: All YOLO scripts expect best.pt at preprocess/a_image_processing/best.pt. Make sure this file exists!
Control Modes:
  • MODE_JOINT — Joint-space control (positions or velocities)
  • MODE_CARTESIAN — Cartesian space control (pose or velocity in task space)
The Cartesian admittance controller handles IK internally when you send pose commands.
Topics Reference:
  • /joint_states — Current joint positions (sensor_msgs/JointState)
  • /aic_controller/joint_commands — Joint commands (JointMotionUpdate)
  • /aic_controller/pose_commands — Cartesian commands (MotionUpdate)
  • /aic_controller/controller_state — Current TCP pose, wrench
  • /aic_controller/change_target_mode — Switch between JOINT/CARTESIAN mode
  • /observations — Observation message (used by policy)
